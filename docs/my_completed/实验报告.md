# 校园导航系统 — 实验报告

> 数据结构课程个人项目：动态图上的校园路线规划与设施分析系统

---

## 一、数据结构设计说明

### 1.1 选型：邻接表 + 双哈希索引

LGraph 内部采用 **两层 `unordered_map`** 的组合结构：

```cpp
// 顶点存储：place_id → 顶点信息
std::unordered_map<std::string, LocationInfo> vertices_;

// 邻接表：from_id → (to_id → 边信息)
std::unordered_map<std::string,
    std::unordered_map<std::string, EdgeNode>> adj_;
```

**设计目标**：
- 按字符串 `place_id` 快速定位顶点和边，避免额外维护 `string→int` 映射层
- 支持高效增删改查（O(1)），适配命令驱动的动态图场景
- 作为无向图，每条边在 `adj_[u][v]` 和 `adj_[v][u]` 各存一份，保证双向查询对称

### 1.2 替代方案与权衡

| 方案 | 顶点查找 | 边查找 | 删顶点 | 空间 | 取舍理由 |
|------|---------|--------|--------|------|----------|
| **邻接矩阵** `vector<vector<int>>` | O(1) | O(1) | O(V) | O(V²) | ❌ 校园图极度稀疏（87节点仅26边），矩阵浪费 99.6% 空间 |
| **纯边表** `vector<EdgeNode>` + map索引 | O(1) | O(E) | O(E) | O(V+E) | ❌ 查特定边需遍历，Dijkstra 每次松弛都查边，O(E) 太慢 |
| **`map`（红黑树）** | O(logV) | O(logV) | O(degree·logV) | O(V+E) | ❌ 本项目不需要按键有序遍历，哈希 O(1) 更快 |
| **双hash邻接表（本方案）** | **O(1)** | **O(1)** | **O(degree)** | **O(V+E)** | ✅ 时空均最优，最适配本项目 |

### 1.3 无向图的对称性保证

- 插入边时同步写入 `adj_[from][to]` 和 `adj_[to][from]`
- 删除边时同步删除两个方向
- 更新边属性（距离/时间/状态）时两个方向同时生效
- `EdgesCount()` 统计时只计 `from_id < to_id` 的方向，避免重复

### 1.4 各操作的时间/空间复杂度

| 操作 | 时间复杂度 | 空间复杂度 |
|------|-----------|-----------|
| `exist_vertex` / `GetVertex` | O(1) | O(1) |
| `InsertVertex` | O(1) | O(1) |
| `DeleteVertex` | O(degree) | O(1) |
| `UpdateVertex` | O(1) | O(1) |
| `exist_edge` / `GetEdge` | O(1) | O(1) |
| `InsertEdge` / `DeleteEdge` | O(1) | O(1) |
| `UpdateEdge` / `CloseRoad` / `OpenRoad` | O(1) | O(1) |
| `AllPlaceIds` | O(V log V) | O(V) |
| `AllEdges` | O(E) | O(E) |
| `GetAdjacentEdges` | O(degree) | O(degree) |
| `GetPlacesByCategory` | O(V) | O(V) |

> **空间总复杂度**：O(V + E)，其中每条无向边存储两份。

## 二、关键算法说明与复杂度分析

### 2.1 A. 连通分量分析（BFS）

- **实现**：对每个未访问的顶点做 BFS，仅沿 `status == "open"` 的边遍历
- **扩展**：支持 `blocked_nodes` 和 `blocked_edges` 参数，用于关键节点/边分析中的"临时屏蔽"
- **时间复杂度**：O(V + E)
  > *每个顶点入队/出队各一次 O(V)，每条无向边被两个方向的 `GetAdjacentEdges` 各遍历一次 O(2E) = O(E)。合计 O(V+E)。*
- **空间复杂度**：O(V)
  > *`visited` 集合 + BFS 队列，最多存储全部 V 个顶点。*

---

### 2.2 B. 最短路径（Dijkstra + 小顶堆）

- **实现**：使用 `priority_queue<pair<int,string>, vector<...>, greater<...>>` 实现小顶堆
- **边权选择**：`getWeight()` 辅助函数根据 `PathMode`（DIST/TIME）选择 `distance` 或 `walk_time`
- **惰性删除**：用 `settled` 集合标记已确定最短路的节点，避免重复处理堆中同一节点的旧记录
- **路径回溯**：`prev` 哈希表记录前驱，从 `to_id` 反向推到 `from_id` 后反转
- **提前退出**：堆顶弹出终点时即可终止（所有边权非负，Dijkstra 贪心性质保证）
- **时间复杂度**：O((V+E) log V)
  > *每个顶点最多被弹出一次（`settled` 去重），每次堆操作 O(log V)，共 O(V log V)；每条边最多触发一次松弛（每个邻居首次被松弛后后续不再入队），每次 push 堆 O(log V)，共 O(E log V)。合计 O((V+E) log V)。*
- **空间复杂度**：O(V)
  > *`dist`、`prev`、`settled` 各 O(V)，优先队列最多容纳 O(E) 个元素。取上界 O(V)。*

---

### 2.3 B'. 时刻约束最短路径

- **实现**：先检查起止点在 `time` 时刻是否开放（`open_time ≤ time ≤ close_time`，HH:MM 字符串字典序天然等于时间序），再跑 Dijkstra，松弛时额外检查邻居的时间窗
- **时间复杂度**：同 Dijkstra O((V+E) log V)
  > *额外的时间窗检查 `isOpen(neighbor)` 是 O(1) 的 `GetVertex` + 两次字符串比较，不增加渐进复杂度。*
- **空间复杂度**：O(V)
  > *同 Dijkstra，额外仅多一个 lambda 捕获引用，无新增空间开销。*

---

### 2.4 C. 必经点路径规划

- **实现**：将路径拆为 `from → w1 → w2 → ... → wk → to`，逐段调用 `GetShortestPath`，拼接时去掉重复衔接点
- **任一段不可达则整体不可达**
- **时间复杂度**：O(k·(V+E) log V)，k 为必经点个数
  > *共 k+1 段，每段跑一次 Dijkstra O((V+E) log V)，拼接和去重是 O(k·V) 的线性操作，被 Dijkstra 主导。合计 O(k·(V+E) log V)。*
- **空间复杂度**：O(V)
  > *每段 Dijkstra 各自独立运行并释放，最终结果 `path` 最多容纳 k·V 个节点。单段上界 O(V)。*

---

### 2.5 D. 最小生成树（Kruskal + DSU）

- **选择理由**：Kruskal 只需对所有边排序一次 O(E log E)，比 Prim 更简洁；DSU 操作近乎 O(1)；稀疏图中 Kruskal 更优
- **实现**：收集所有 open 边按 distance 升序排序 → 给每个地点分配整数编号 → 逐边用 DSU 判断是否形成环 → 不形成环则加入 MST → 最终检查 MST 边数是否 = V-1（否则图不连通）
- **DSU 优化**：路径压缩 + 按秩合并，摊还复杂度 O(α(n)) ≈ O(1)
- **时间复杂度**：O(E log E)
  > *收集 open 边 O(E)；排序 O(E log E)；逐边 DSU 判断 O(E·α(V)) ≈ O(E)；建 `id_to_idx` 映射 O(V)。主导项为排序 O(E log E)。*
- **空间复杂度**：O(V + E)
  > *排序的 `open_edges` 数组 O(E)，DSU 的 `parent`/`rank_` 数组 O(V)，`id_to_idx` 映射 O(V)。合计 O(V+E)。*

---

### 2.6 E. 关键节点与关键边分析（暴力枚举法）

- **实现**：先算 baseline 连通分量数 → 逐点"屏蔽"后重算分量数（> baseline 则为关键节点）→ 逐条 open 边"屏蔽"后重算分量数（> baseline 则为关键边）
- **屏蔽实现**：通过 BFS 的 `blocked_nodes` 和 `blocked_edges` 参数实现"临时删除"，无需修改原图
- **时间复杂度**：O(V·(V+E) + E·(V+E))
  > *baseline BFS 一次 O(V+E)。对每个顶点 v，屏蔽后重跑 `bfsComponents` O(V+E)，共 V 次 → O(V·(V+E))。对每条 open 边 e，屏蔽后重跑 `bfsComponents` O(V+E)，共 E 次 → O(E·(V+E))。合计 O(V·(V+E) + E·(V+E))。*
- **空间复杂度**：O(V+E)
  > *每次 `bfsComponents` 独立调用、独立释放。单次需要 `visited` O(V) + 队列 O(V) + `blocked_nodes`/`blocked_edges` O(V+E)。取上界 O(V+E)。*
- **千节点规模下可接受**：实测 chain_1000（1000 节点）可正常运行
- **加分项展望**：Tarjan 算法可一遍 DFS 同时求割点和桥，复杂度 O(V+E)

### 2.7 算法复杂度汇总

| 算法 | 时间复杂度 | 空间复杂度 |
|------|-----------|-----------|
| A. 连通分量 (BFS) | O(V+E) | O(V) |
| B. 最短路径 (Dijkstra) | O((V+E)logV) | O(V) |
| B'. 时刻约束最短路径 | O((V+E)logV) | O(V) |
| C. 必经点路径 | O(k·(V+E)logV) | O(V) |
| D. 最小生成树 (Kruskal) | O(E log E) | O(V+E) |
| E. 关键节点/边 (暴力枚举) | O(V·(V+E)+E·(V+E)) | O(V+E) |

## 四、遇到的问题与解决思路

### 4.1 无向图边的双向一致性

**问题**：增删改边时，两个方向的邻接表必须同步更新，容易遗漏。

**解决**：`InsertEdge`、`DeleteEdge`、`UpdateEdge` 中统一处理双向。`CloseRoad`/`OpenRoad` 复用 `UpdateEdge`，避免重复代码。

### 4.2 删除顶点的级联清理

**问题**：删除顶点后，与之相邻的节点邻接表中仍残留指向被删顶点的边记录。

**解决**：`DeleteVertex` 中先遍历被删顶点的所有邻边，在邻居的 `adj_` 中也 `erase` 掉对应的边，再删除顶点本身。

### 4.3 Dijkstra 的"惰性删除"

**问题**：松弛时向优先队列 push 了新记录，但旧记录仍留在队列中。如果直接出队处理，会导致重复处理。

**解决**：用 `settled` 集合记录已确定最短路的节点。每次 `pop()` 后先检查是否在 `settled` 中，若是则 `continue` 跳过。

### 4.4 关键边分析中的同权边问题

**问题**：暴力枚举每条 open 边时，若图中存在等权边，`AllEdges` 返回边的顺序影响 BFS 的遍历顺序，可能导致分析结果的不确定性。

**解决**：BFS 遍历邻接边时仅判断该边是否在 `blocked_edges` 中，不依赖边权顺序，保证了每轮关键边分析的确定性。

### 4.5 命令行输出排序规则

**问题**：`QUERY_CATEGORY`、`ADJ`、`MST`、`CRITICAL` 等命令的输出有严格的排序要求（字典序/降序），容易遗漏。

**解决**：在 CommandProcessor 的每个 cmd 函数中统一 `std::sort` 后再输出，不在底层数据提供函数中硬编码排序。

## 五、AI 协作记录

### 5.1 关键提示词对话（精选 6 条）

#### 对话 1：LGraph 数据结构选型探讨

**场景**：骨架代码只给了接口，我需要自己设计内部存储结构，拿不定主意。

**我的提示词**：
> LGraph 内部用什么数据结构比较好？项目有 1000 节点，需要快速按 place_id 查顶点、快速增删边、Dijkstra 遍历邻接表。

**AI 回应要点**：
> 推荐 `unordered_map<string, LocationInfo>` + `unordered_map<string, unordered_map<string, EdgeNode>>`。空间 O(V+E)，增删改查 O(1)。然后用对比表分析了邻接矩阵和纯边表的优缺点。

**我的处理**：接受。这是核心设计决策，AI 的对比分析帮我清晰地理清了各方案的权衡。

---

#### 对话 2：Dijkstra 小顶堆的实现方式

**场景**：我知道 Dijkstra 需要小顶堆，但 `priority_queue` 默认是大顶堆。

**我的提示词**：
> priority_queue 怎么变成小顶堆？元素是 `pair<int, string>`。

**AI 回应要点**：
> 传入 `std::greater<pair<int,string>>` 作为第三个模板参数，比较时先比 first 再比 second。同时介绍了惰性删除的模式。

**我的处理**：部分采纳。小顶堆方案完全采用；但惰性删除的具体实现我做了调整——用 `unordered_set<string> settled` 而非 AI 建议的 `vector<bool> visited`，因为我用的是字符串 ID。

---

#### 对话 3：CsvIO 解析策略

**场景**：CSV 用逗号分隔，但 `istringstream >>` 天然按空格分割。

**我的提示词**：
> 怎么用 `istringstream` 解析逗号分隔的 CSV？

**AI 回应要点**：
> 先 `replace(line.begin(), line.end(), ',', ' ')` 把逗号全换空格，然后直接 `iss >> field1 >> field2 >> ...`。

**我的处理**：接受。但 AI 给了 `std::replace`，我改成了遍历自己替换（`for (char &c : result)`），因为我只需要处理一行，不需要额外引入 `<algorithm>` 的复杂性。

---

#### 对话 4：C++17 结构化绑定语法

**场景**：在 Dijkstra 中取 `pq.top()` 返回的 `pair<int, string>`，写 `p.first`/`p.second` 觉得不够直观。

**我的提示词**：
> C++ 有没有办法把 pair 直接解包成两个变量？

**AI 回应要点**：
> C++17 支持结构化绑定：`auto [a, b] = pq.top()`。需要 `-std=c++17`。

**我的处理**：完全接受。`auto [cur_dist, cur_id] = pq.top()` 大幅提升了代码可读性。

---

#### 对话 5：Lambda 捕获列表的 `[&]` 含义

**场景**：代码中用了 `[&](const string& a, const string& b) { ... }`，我最初不太理解这个语法。

**我的提示词**：
> `[&]` 在 lambda 中是什么意思？

**AI 回应要点**：
> `[&]` 是捕获列表，表示以引用方式捕获所有外部变量。效率高不拷贝，但要注意生命周期。还给出了 `[=]` 和 `[&x, =y]` 的变体。

**我的处理**：接受。理解后我在代码中灵活使用了 `[&]` 来访问外部 `blocked_edges`、`graph` 等变量。

#### 对话 6：`emplace_back` vs `push_back`

**场景**：看到代码里有的地方用 `push_back`，有的用 `emplace_back`。

**我的提示词**：

**我的处理**：接受。后续代码中构造 `LocationInfo` 时全部改用 `emplace_back`。

### 5.2 调试故事（精选 2 个）

#### Bug 1：`CmdLoad` 中 LOAD 多次后边数翻倍

**现象**：连续调用两次 `LOAD places.csv roads.csv`，图的边数翻倍。

**诊断**：`cmdLoad` 没有清空旧图，导致新数据叠加在旧图上。

**AI 建议**：在 LOAD 前先调用 `graph.Clear()` 清空旧图。

**我的处理**：接受。我先是手动 `DeleteVertex` 逐个删，AI 指出不如直接加一个 `Clear()` 方法。采纳后我在 `LGraph` 中实现了 `Clear()`（直接 `vertices_.clear(); adj_.clear()`），然后在 `cmdLoad` 中一行 `graph.Clear()` 替代了之前的 10 行清空代码。

#### Bug 2：BFS 在屏蔽节点时连邻居也一起跳过

**现象**：关键节点分析中，屏蔽节点 P0001 后，其所有邻居也被跳过了，分析结果错误。

**诊断**：BFS 内部 `if (blocked_nodes.count(cur)) continue` 写在了 `while (!q.empty())` 的最前面。这导致从任何节点遍历到被屏蔽节点的邻边都会提前结束。

**AI 建议**：把屏蔽检查移到遍历邻边的循环内部，只跳过对被屏蔽节点的访问，不跳过当前节点本身。

**我的处理**：接受。重构了 `bfsComponents`，屏蔽节点的检查放在遍历邻边时（`if (blocked_nodes.count(neighbor)) continue`），不放在弹出节点后。

### 5.3 拒绝过的 AI 建议（1 条）

**场景**：AI 建议在 `GetEdge` 中复用 `exist_edge` 来检查边是否存在。

**AI 建议**：
```cpp
EdgeNode LGraph::GetEdge(const string &from, const string &to) const {
    if (!exist_edge(from, to))
        throw GraphException("road_not_found");
    return adj_.at(from).at(to);
}
```

**我的拒绝理由**：这样会导致两次哈希查找——`exist_edge` 中 `adj_.find(from)` 一次，`GetEdge` 中 `adj_.at(from)` 又一次。对于高频调用的函数（Dijkstra 每次松弛都要调），这是不必要的性能浪费。

**我的替代方案**：
```cpp
auto from_it = adj_.find(from_id);
if (from_it == adj_.end()) throw ...;
auto to_it = from_it->second.find(to_id);
if (to_it == from_it->second.end()) throw ...;
return to_it->second;
```
一次查找拿到迭代器后直接访问数值，O(1) 但不重复。

## 五、拓展任务设计与实现

### 5.1 拓展 1：共享单车 SHORTEST_K（9 分）

**问题**：给定起点、终点和 K 张加速券，每张券可将一条边的 walk_time 缩短为 ceil(walk_time / 3)，求最短到达时间。

**算法思路——二维 Dijkstra**：常规 Dijkstra 的状态是"走到哪个节点"。拓展 1 的核心变化是把"已经用了几张券"也纳入状态维度。状态编码为 `"place_id|used"`（用 `|` 拼接成字符串键），在一个统一的优先队列里管理 O(K·V) 个状态。对每条邻边分两种情况扩展：不骑车（cost += walk_time，used 不变）和骑车（cost += ceil(walk_time / 3)，used + 1）。所有可能性在小顶堆里竞争，首次到达终点时的状态就是全局最优。

**路径回溯**：维护 `prev[key] = {prev_id, used, used_bike_on_this_edge}`。从终态沿 prev 链回溯，path 收集节点（reverse），fast_edges 收集用券边（字典序排序）。

**复杂度**：O(K·(V+E) log(K·V))。K ≤ 10 时与普通 Dijkstra 同阶。

**验证**：拓展 1 small case 与 answer.txt 逐字一致（已通过 fc /w 验证）。
